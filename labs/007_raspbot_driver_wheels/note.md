# How an SMBus write controls a Raspbot V2 wheel

The central idea is that an SMBus write does not continuously power a wheel. It sends a short configuration command to the onboard MCU. After receiving that command, the MCU's hardware PWM timer continues driving the motor while the Raspberry Pi and Linux are doing other work—or even sleeping.

## Complete control path

The Rust program takes this short route:

```text
Rust program
  ↓ ioctl through /dev/i2c-1
Raspberry Pi I²C controller
  ↓ SDA + SCL wires
Raspbot MCU at address 0x2B
  ↓ direction GPIO + PWM output
H-bridge motor driver
  ↓ switched battery voltage and current
DC gearmotor
  ↓
Mecanum wheel
```

The ROS system takes a longer route but eventually reaches the same lower level:

```text
Remote application
  ↓ WebSocket JSON
rosbridge_suite on the Pi
  ↓ ROS topic /cmd_vel
ROS motor-driver node
  ↓ mecanum wheel calculations
Four signed wheel commands
  ↓ Raspbot I²C API
MCU → H-bridge → motors
```

The Rust program bypasses the network, WebSocket, ROS, `/cmd_vel`, and mecanum calculations. It chooses each wheel directly.

## 1. The command created by the Rust program

The MCU's 7-bit I²C address is:

```rust
const RASBOT_I2C_ADDRESS: u16 = 0x2B;
```

For motor zero, forward, maximum requested speed, the program creates:

```text
command   motor ID   direction   speed
  0x01      0x00       0x00      0xFF
```

The Rust code constructs the three data bytes and sends them with command `0x01`:

```rust
let data = [
    motor_info.id as u8,
    motor_info.direction as u8,
    motor_info.speed,
];

self.write_array(0x01, &data)?;
```

The API calls `0x01` a register, but it is probably better understood as a firmware command selector:

```text
0x01 = motor command
0x02 = servo command
0x03 = all LEDs
0x04 = individual LED
...
```

It is not necessarily a literal hardware register in the MCU.

## 2. What `LinuxI2CDevice::new()` does

This opens the Linux interface to the Pi's physical I²C controller:

```rust
LinuxI2CDevice::new("/dev/i2c-1", 0x2B)
```

It effectively tells Linux:

```text
Use I²C controller 1.
Address future operations to slave 0x2B.
```

No motor power is controlled at this point.

When the program calls:

```rust
smbus_write_i2c_block_data(0x01, &[0, 0, 255])
```

the Rust library asks the Linux kernel to perform an I²C transaction. The Pi's I²C hardware generates the electrical signals.

## 3. What appears on SDA and SCL

I²C uses:

- `SCL`: clock generated by the Pi
- `SDA`: data driven by the Pi or MCU
- A shared ground
- Pull-up resistors that normally bring SDA and SCL to approximately 3.3 V

For motor zero, forward, speed 255, a logic analyzer should decode approximately:

```text
START
Address 0x2B + Write
ACK from MCU
0x01    command: motor control
ACK
0x00    motor ID 0
ACK
0x00    forward
ACK
0xFF    requested speed 255
ACK
STOP
```

Some analyzers display the address as `0x56`. This is the same address represented as the complete 8-bit address byte:

```text
0x2B << 1 | write_bit = 0x56
```

The entire transaction takes much less than one millisecond.

An `Ok(())` result means the I²C operation completed and was acknowledged. It does not prove that the physical motor rotated. A disconnected motor or failed H-bridge can still produce a successful I²C result.

## 4. What the MCU probably does

The exact Yahboom firmware is proprietary, but its receive logic must be roughly equivalent to:

```c
void on_i2c_write(uint8_t *bytes, size_t length) {
    uint8_t command = bytes[0];

    if (command == 0x01 && length >= 4) {
        uint8_t motor_id = bytes[1];
        uint8_t direction = bytes[2];
        uint8_t speed = bytes[3];

        set_motor_direction(motor_id, direction);
        set_motor_pwm(motor_id, speed);
    }
}
```

The Python driver's test comments identify the logical motors as:

```text
ID 0 → L1
ID 1 → L2
ID 2 → R1
ID 3 → R2
```

L1/L2 and R1/R2 are likely the front/rear pairs, but their exact physical positions should be verified experimentally.

The MCU changes two kinds of output:

- Direction logic sent to the H-bridge
- A PWM signal whose duty cycle is derived from `speed`

A simple mapping would be:

```text
PWM duty ≈ speed / 255
```

For example:

```text
speed 0   → 0% duty
speed 64  → approximately 25%
speed 128 → approximately 50%
speed 255 → approximately 100%
```

This mapping is probable rather than confirmed. The firmware may impose a minimum duty, ramp acceleration, invert individual motors, or remap the values.

## 5. Why the MCU and H-bridge are separate

The MCU cannot supply motor current. Its pins only produce low-power logic signals.

The motor-current path is:

```text
Battery positive
  ↓
H-bridge transistor
  ↓
Motor winding
  ↓
Opposite H-bridge transistor
  ↓
Battery negative
```

A simplified H-bridge looks like:

```text
             Battery +
              │     │
             Q1     Q2
              │     │
              ├─ Motor ┤
              │     │
             Q3     Q4
              │     │
             Ground
```

One diagonal transistor pair drives current in one direction:

```text
Q1 + Q4 → current flows one way
```

The other diagonal reverses it:

```text
Q2 + Q3 → current flows the other way
```

PWM rapidly turns part of that path on and off. At 50% duty, the motor normally receives full battery voltage in pulses for approximately half the time—not a perfectly steady half-battery voltage. Motor inductance and mechanical inertia smooth the result.

## 6. Current, torque, and speed

The `speed` byte is not a direct current command and probably is not a measured RPM command. It is most likely an open-loop PWM request.

Motor electrical behavior is approximately:

```text
V = L(di/dt) + Ri + Keω
```

Where:

- `V` is applied motor voltage
- `L` is winding inductance
- `i` is motor current
- `R` is winding resistance
- `Keω` is the back-EMF produced while spinning

Motor torque is approximately:

```text
torque = Kt × current
```

Consequences:

- At startup, speed and back-EMF are zero, so current can be large.
- Under mechanical load, the motor slows, back-EMF falls, and current rises.
- At stall, current can become very large.
- PWM controls average applied voltage indirectly.
- Actual current depends on the motor, battery, H-bridge, load, and speed.

Therefore, `speed = 128` does not mean 128 RPM or half the current. It means something closer to "request approximately half PWM duty."

## 7. Why the motor keeps running during `sleep()`

The program sends four short I²C transactions:

```text
motor 0 = forward, 255
motor 1 = forward, 255
motor 2 = forward, 255
motor 3 = forward, 255
```

The MCU retains these states and its hardware PWM timers continue autonomously. Linux does not need to send `255` repeatedly.

The program then sleeps for ten seconds. After the delay it sends:

```text
[motor ID, backward, 0]
```

Because PWM is zero, the direction value is irrelevant. The precise stop behavior depends on the proprietary MCU firmware and H-bridge:

- **Coast:** the H-bridge disconnects the motor
- **Brake:** the H-bridge electrically brakes the motor

The four motors are also not started at exactly the same instant. They receive four separate I²C transactions, probably separated by only hundreds of microseconds, so they appear to start simultaneously mechanically.

## 8. Where ROS fits

The ROS bridge publishes a `geometry_msgs/Twist` message on `/cmd_vel`:

```text
linear.x  = forward/backward velocity
linear.y  = sideways velocity
angular.z = rotation
```

A mecanum driver converts these three requested motions into four wheel commands. One common arrangement is:

```text
front-left  = forward - sideways - rotation
rear-left   = forward + sideways - rotation
front-right = forward + sideways + rotation
rear-right  = forward - sideways + rotation
```

The exact signs depend on wheel roller orientation, wiring, motor placement, and coordinate conventions.

The driver then normalizes the four values and converts them into the Yahboom format:

```python
for motor_id, signed_speed in enumerate(wheel_speeds):
    direction = 1 if signed_speed < 0 else 0
    speed = min(abs(signed_speed), 255)
    Ctrl_Car(motor_id, direction, speed)
```

That final call sends the same `0x01` I²C command implemented by the Rust program.

The actual `Mcnamu_driver` implementation is not present in this lab, so this ROS-to-I²C final step is an architectural inference rather than something fully proven by the current files.

## 9. Sensor data travels mostly upward

Motor control flows downward:

```text
Software → MCU → H-bridge → motor
```

Sensor information flows upward:

```text
Sensor → MCU or Pi driver → ROS topic → rosbridge → remote application
```

The existing Python library shows:

- Tracking sensors read from location `0x0A`
- Ultrasonic distance read from locations `0x1A` and `0x1B`
- Infrared remote value read from location `0x0C`

Sensors do not automatically control the motors. An avoidance or navigation program must explicitly connect sensing to movement:

```text
Sensor detects obstacle
  ↓
Navigation algorithm makes a decision
  ↓
Algorithm publishes /cmd_vel
  ↓
Motor driver calculates wheel commands
  ↓
I²C motor commands
```

ROS transports messages; it does not decide how the robot should move.

## 10. Confirmed facts versus unknown internals

The working local driver directly confirms:

- Raspberry Pi I²C bus 1
- MCU I²C address `0x2B`
- Motor command `0x01`
- Four motor IDs
- Direction byte `0` or `1`
- Requested magnitude byte `0–255`
- Sensors and peripherals exposed through other MCU commands or register-like locations

The following remain unknown:

- Exact MCU model and firmware
- Exact MCU pins connected to each motor driver
- Exact H-bridge model
- PWM frequency
- Exact mapping from `0–255` to duty cycle
- Coast versus brake behavior at zero
- Whether any current limiting or acceleration ramp is implemented

The additional hardware and sensor notes mix information from Raspbot V2, Yahboom G1/ROSMASTER, and Yahboom's separate ESP32-S3 Micro-ROS board. Claims about encoders, `/odom`, an onboard IMU, USB-UART, or Micro-ROS should be verified on the physical robot before being treated as facts.

## 11. How to observe each layer

A logic analyzer connected to SDA, SCL, and ground can reveal:

- Address `0x2B`
- Command `0x01`
- Motor ID
- Direction
- Requested speed
- ACK/NACK behavior

An oscilloscope or logic analyzer connected to the H-bridge logic inputs can reveal:

- Which MCU output controls each motor
- PWM frequency
- Actual duty-cycle mapping
- Direction truth table
- Coast versus brake behavior

An oscilloscope across a motor terminal pair can reveal the switched motor voltage, but this measurement must be made carefully because neither terminal is necessarily at ground. A differential probe is the safe measurement method.

The current path must be measured with an appropriate current probe or a correctly sized current-sense resistor. The Raspberry Pi and MCU never directly carry the motor current.
